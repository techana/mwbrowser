**Author:** discussion captured during file_load_architecture session

## What we're trying to solve

Today's render path:

1. Bridge sends HTML chunks via serial.
2. MSX parser walks every byte, dispatches tags, emits text glyphs to
   a line buffer, flushes lines to VRAM.
3. Each scroll triggers a full re-render ŌĆö the parser walks from byte
   0 of the buffer to the target line, even if only line offsets
   changed.

Cost on the bench page (BENCHTX, ~12 KB):

- Initial render: ~8 s wall clock.
- Each PageDown / PageUp: another ~8 s (full re-walk).

The slideshow projects (`MSX New Font Demo`, `MSX1 Arabic Slideshow`,
`MSX Arabic Slideshow`) put full screen images on screen in
*milliseconds* by streaming raw VRAM bitmap data from disk and blitting
it via the V9938's HMMC / OUT-tight loop. Could we apply the same
trick to the HTML browser?

## The basic idea

Move HTML rendering off the MSX entirely. The bridge:

1. Headlessly renders the page to an image at the MSX viewport size
   (492 ├Ś ~600 px tall, depending on how much scroll the page needs).
2. Encodes the rendered pixels as Screen-6-format bytes (2 bpp,
   palette-quantised ŌĆö we already do this for inline `<img>` tags via
   `tools/web_bridge.py`'s PCX encoder).
3. Sends the bitmap, possibly RLE-compressed, over the wire.
4. The MSX blits it directly to VRAM.

The on-MSX renderer (`RenderSc6File`, ~286 B) already knows how to
load Screen-6 bitmaps; the disk path that consumes `PGnn.SC6` files
generated by `tools/web_to_sc6.py` is the existing local-disk version
of this idea. The serial-side equivalent is a new bridge command that
streams the same bytes over the wire.

## Bandwidth budget

One Screen-6 viewport = 512 ├Ś 212 ├Ś 2 bits/pixel = **27 KB raw**.
With RLE / pack-bits (Screen 6 has lots of flat-colour runs),
typically **~12 KB compressed**.

Wire transfer time (raw / compressed):

| Wire rate                                | Raw 27 KB | RLE 12 KB |
|------------------------------------------|----------:|----------:|
| 19200 baud  (real-cartridge ceiling)     |    ~14 s  |    ~6 s   |
| 38400 baud  (good cartridges)            |     ~7 s  |    ~3 s   |
| 115200 baud (patched openMSX / future HW)|    ~2.4 s |    ~1 s   |

Compare to today's HTML-streaming + render path on BENCHTX:

- Bridge transfer (12 KB) @ 115200: ~3 s.
- MSX render: ~8 s.
- Total first load: ~11 s.
- Each subsequent scroll: another ~8 s of full re-render.

So bitmap mode beats us by ~7 s on first load AND ~5 s on every
scroll, *at 115200 baud*. At 19200 baud the first-load is roughly a
wash but bitmap mode still wins big on every scroll because there's
no re-walk.

The slideshow project hits "milliseconds" because the .S02 lives on
the disk image ŌĆö local FDD I/O is ~50 KB/s, much faster than any
serial wire. Our equivalent local fast path is `RenderSc6File`
loading from `A:PGnn.SC6`; that's already milliseconds when the file
is on disk. The bridge-side equivalent over serial is wire-bound and
will not match disk speed.

## What the bitmap pipeline gives up

- **Click-on-link** stops working unless the bridge also sends a
  side-metadata blob: `[(rect_x1, rect_y1, rect_x2, rect_y2, url), ŌĆ”]`.
  The on-MSX click dispatch already keeps a `LinkUrls` table; the
  bridge just needs to populate it directly via a sidecar instead of
  the parser building it from `<a href>` tags.
- **Form input** ŌĆö same fix: rect + (name, type, current value) sidecar
  per form widget. The MSX-side rendering of a focused widget can
  overlay the bitmap with a simple cell-aligned text box.
- **Anchor links / `#section`** ŌĆö needs a `(target ŌåÆ scroll_y_pixels)`
  lookup table.
- **Highlight / cursor / selection** ŌĆö drawn on top of the bitmap,
  separately. We already do this for the link-hover cursor.

None of these are show-stoppers. They're all "send a small metadata
blob alongside the bitmap."

## Three plausible architectures

### A. Full-doc bitmap

Bridge renders the entire scrolled height of the page into one tall
bitmap (e.g. 512 ├Ś 1500 px). MSX downloads it once, stashes it in VRAM
(Screen 6 has ~64 KB usable VRAM per page; multiple pages give us
~256 KB). Scroll = adjust the VRAM read base register.

- **Pros:** instant scroll after the initial transfer. Real "set R23
  to scroll" hardware-accelerated panning.
- **Cons:** caps the doc at one VRAM-region-worth of pixels (~600 px
  tall = a few screens). Long Wikipedia articles overflow.

### B. Incremental viewports over the wire

Bridge sends one bitmap per viewport on demand: `GET VIEW <scroll_y>`.
Scroll = request the next viewport, blit when it arrives.

- **Pros:** unbounded doc length.
- **Cons:** every scroll round-trips through the wire (~1ŌĆō3 s at
  115200, ~6 s at 19200). Still much faster than today's re-render
  for big pages, but not "instant."

### C. Hybrid

Keep the HTML-streaming path for **interactive pages** (forms,
search, link-heavy content where on-MSX layout matters). Add a
**BITMAP / VIEW mode** for **content-heavy view-only pages**
(Wikipedia article, blog post, news). User toggles, or the bridge
picks heuristically (e.g. detects "no forms, low link density ŌåÆ
bitmap mode").

This is the recommended starting architecture. It preserves
everything we have, adds a viewer mode for the cases where it pays.

## Implementation sketch (when we pick this up)

### Bridge (`tools/web_bridge.py`)

- New `MsxSession.handle_get` branch for `BITMAP <url> <scroll_y>`
  (or `VIEW <offset>` once we know the URL is in session state).
- Reuse the headless-render path the bridge already uses for image
  proxying (Pillow / wkhtmltopdf / Playwright).
- Quantise pixels to the Screen-6 4-colour palette (already done for
  inline `<img>` PCX encoding ŌĆö same `_to_pcx` lookup).
- RLE-pack rows; emit `OK BMP <bytes> <chunk>/<total>\r\n<body>` ŌĆö
  same wire frame as the existing `OK HTM` paginated format, just a
  different `<kind>`.
- Sidecar metadata: `OK MAP <bytes>\r\n<body>` carrying
  `[link_count, (x1, y1, x2, y2, url_len, url_bytes) * link_count]`.

### MSX (`src/mwbrowser.asm`)

- New `SerialKind = 3` (= BMP). `RemoteGet` already parses kind
  bytes; extend the dispatch.
- A new `RenderRemoteBitmap` routine that streams body bytes
  straight into VRAM via OUTI loops (no parsing, no buffering past
  the chunk boundary). Re-uses the OUTI tightness from
  `RenderSc6File`.
- A new `LinkRectsFromMap` that decodes the sidecar into the
  existing `LinkStartX/Y/EndX/Y/Urls` arrays so click dispatch keeps
  working.
- Browser mode flag (`Mode: db 0` ŌĆö 0 = HTML, 1 = BMP). Toolbar
  toggle button. Initial pick: HTML for forms, BMP for everything
  else.

### Estimated work

- ~500 B of new MSX code (RenderRemoteBitmap + map decoder + mode
  flag wiring).
- ~150 lines of bridge Python (BITMAP handler + headless render +
  quantise + RLE).
- A one-page benchmark showing HTML mode vs BMP mode on a real
  Wikipedia article at 19200, 38400, 115200 baud.
